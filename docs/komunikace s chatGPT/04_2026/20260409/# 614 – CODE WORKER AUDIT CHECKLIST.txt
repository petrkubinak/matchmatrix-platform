# 614 – CODE WORKER AUDIT CHECKLIST

Datum: 2026-04-09

## 1. Cíl

Cílem tohoto checklistu je přepnout MatchMatrix z:
- DB/OPS připravenosti

na:
- skutečnou runtime připravenost pro každý sport a entitu

Neřešíme teď kvalitu dat.
Řešíme existenci a úplnost execution layer.

---

## 2. Základní princip

Sport je runtime ready pouze pokud pro danou entitu existuje:

1. worker
2. ingest script / runner
3. parser / transform
4. merge flow do public vrstvy
5. OPS napojení
6. run/test způsob spuštění
7. audit / health check

Pokud některý bod chybí, sport/entity není runtime ready.

## 5. Povinné kontrolní body pro každou sport × entity kombinaci

### A. Worker existence
- Existuje dedikovaný worker v `workers\`?
- Má jasný naming?
- Je sport-specific nebo reusable template?
- Umí logovat run_id / provider / sport / entity?
- Umí fail-fast při chybě configu?

### B. Ingest entrypoint
- Existuje spouštěcí `.py` nebo `.ps1`?
- Je jasné, jak se spouští ručně?
- Je jasné, jak se spouští z OPS/planneru?
- Má vstupní parametry provider / sport / entity / run_group?

### C. Provider pull layer
- Existuje pull script pro provider?
- Umí uložit RAW payload?
- Umí retry / timeout?
- Umí page/season/league loop?
- Umí budget-safe režim?

### D. Parser / staging layer
- Existuje parser do `staging.stg_provider_*`?
- Má sport-specific mapping?
- Umí canonical provider keys?
- Umí logovat counts: fetched / parsed / skipped / failed?
- Má ochranu proti duplicitám?

### E. Merge / public layer
- Existuje merge SQL nebo merge runner?
- Je znám cílový public objekt?
- Umí provider_map update?
- Umí alias sync?
- Umí safe upsert?
- Umí rerun bez rozbití dat?

### F. OPS binding
- Existuje vazba do:
  - `ops.provider_sport_matrix`
  - `ops.ingest_entity_plan`
  - `ops.provider_entity_coverage`
  - `ops.ingest_targets`
  - `ops.ingest_planner`
  - `ops.provider_jobs`
- Je entita vidět ve view pro orchestraci?
- Je možné ji zařadit do queue?

### G. Runtime control
- Existuje healthcheck?
- Existuje smoke test?
- Existuje debug SQL pro poslední běh?
- Existuje logický stav:
  - STRUCTURE_READY
  - PROVIDER_LIMITED / ACTIVE_PROVIDER
  - EXECUTION_MISSING / EXECUTION_READY

### H. Reusability
- Dá se worker pattern použít pro další sporty?
- Co je hardcoded na football?
- Co musí být parametrizované?
- Co má být v shared helper vrstvě?

---

## 6. Výstup auditu pro každý řádek

Každá sport × entity kombinace musí skončit jedním výsledkem:

### READY
Existuje kompletní execution chain a lze spustit.

### PARTIAL
Existuje část execution chain, ale chybí některé klíčové části.

### OPS_ONLY
OPS a DB existují, ale runtime neexistuje.

### DESIGN_ONLY
Existuje návrh nebo struktura, ale není implementace.

### BLOCKED_PROVIDER
Execution lze postavit, ale provider je limitovaný nebo nevhodný.

## 12. Auditní rozhodovací pravidla

### Worker exists = ANO
Pouze pokud:
- soubor skutečně existuje
- je spustitelný
- patří ke konkrétní entitě
- není to jen experiment nebo starý zbytek

### Parser exists = ANO
Pouze pokud:
- skutečně plní staging/public flow
- není to jen raw fetch bez transformace

### Merge exists = ANO
Pouze pokud:
- existuje jasný merge do public vrstvy
- ne pouze staging insert

### OPS binding exists = ANO
Pouze pokud:
- worker je skutečně napojen do OPS logiky, ne jen ručně spustitelný
