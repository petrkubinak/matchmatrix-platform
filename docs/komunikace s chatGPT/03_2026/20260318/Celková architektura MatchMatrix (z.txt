Z toho, co jsem zatím viděl, je projekt architektonicky velmi dobrý, jen je potřeba dotáhnout pár pipeline.

To, co ukazuješ, potvrzuje, že jsi udělal přechod z fotbalově-specifického modelu na unified multisport model, což je přesně to, co dává pro MatchMatrix smysl. Ve stromu projektu je pořád vidět starší větev staging.api_football_*, ale zároveň už máš zavedenou novou větev staging.stg_provider_* a orchestrace v ops.*.

Jak to dnes funguje jako celek

1. OPS vrstva rozhoduje, co se má stáhnout
Na OPS diagramu jsou vidět tabulky jako ingest_entity_plan, ingest_targets, ingest_planner, ingest_planner_queue, scheduler_queue, provider_jobs, api_budget_status, worker_locks a dashboard view. To znamená, že systém už má vlastní řídicí vrstvu pro plánování ingestu, throttling a běhy workerů. Ta vrstva už není jen pro football.

2. Ingest cycle spouští planner → extractor → merge
run_ingest_cycle_v3.py dělá přesně tento tok:

získá lock v ops.worker_locks

zapíše audit do ops.job_runs

spustí run_ingest_planner_jobs.py

spustí extract_teams_from_fixtures_v2.py

spustí run_unified_staging_to_public_merge_v3.py

zapíše výsledek a uvolní lock.

To je důležité: hlavní orchestrace už je postavená nad unified stagingem, ne nad api_football_*.

3. Unified staging je teď skutečné centrum ingestu
Na staging diagramu je vidět, že dnes používáš hlavně:

stg_api_payloads

stg_provider_leagues

stg_provider_teams

stg_provider_fixtures

stg_provider_players

stg_provider_player_stats

stg_provider_odds

stg_provider_events

Vedle toho tam stále zůstávají starší tabulky jako api_football_leagues, api_football_fixtures, api_football_teams, api_football_odds. To potvrzuje tvou poznámku: začal jsi football-specific modelem, ale pak ses správně přesunul na providerově neutrální staging.

4. Merge do public už bere data z unified stagingu
run_unified_staging_to_public_merge_v3.py čte přímo z:

staging.stg_provider_leagues

staging.stg_provider_teams

staging.stg_provider_players

staging.stg_provider_fixtures

a promítá je do canonical tabulek:

public.leagues

public.league_provider_map

public.teams

public.league_teams

public.players

public.player_provider_map

public.matches.

To je přesně ten správný směr pro multi-sport platformu.

5. Nad public vrstvou běží analytika a denní pipeline
run_daily_pipeline.py pak spouští:

compute_mmr_ratings.py

build match features přes SQL task

predict_matches_V3.py

Co z toho plyne k tvé otázce / poznámce

Ano, api_football_* už by dnes měla být spíš legacy / přechodová větev, ne hlavní budoucnost systému.

Dává mi smysl rozdělit to takto:

Ponechat jako hlavní

Brát jako legacy nebo dočasné

To odpovídá i tomu, co vidíme ve stromu projektu: novější SQL a worker logika už se točí kolem unified_staging, ops provider jobs, planneru a multisport seedů.

Kde je teď slabé místo

Slabina není v tom, že bys přešel na stg_provider_*.
Naopak — to byl správný krok.

Slabé místo je, že systém je teď částečně mezi dvěma světy:

starší football tabulky stále existují

některé workery nebo pomocné skripty na ně ještě sahají

ale hlavní merge už jede přes unified staging

To je přesně důvod, proč ti repair skript na api_football_teams nedával architektonicky smysl.

Můj závěr

Tvoje cílová architektura má být jednoznačně postavená na stg_provider_*, ne na api_football_*.
Football-specific tabulky bych dnes bral jen jako:

historickou stopu,

pomocný importní mezikrok,

nebo kandidáta na úplné vyčištění obsahu při resetu.

A pokud chceš být připravený na měsíc s API PRO a pak rozšířit ingest na více sportů, tak hlavní osa má být:

OPS planner → provider jobs → stg_provider_* → public.* → analytics

To už v tvém kódu i DB dnes reálně je.

Další logický krok je teď udělat čisté rozhodnutí:
co ještě ponechat z api_football_* a co už definitivně považovat za legacy.
Pak ti z toho můžu sestavit přesný plán „co zachovat / co vyprázdnit / co už nepoužívat“


Ano — tohle už je velmi čitelné.

Ty dva panely dávají smysl a dohromady tvoří správný provozní model:

Mission Control / Audit panel = dohled, audit, stav projektu, orientace

1️⃣ Nejdůležitější tabulka pro limit API
ops.api_budget_status
CREATE TABLE ops.api_budget_status (
    id bigint NOT NULL,
    sport_code text NOT NULL,
    request_day date DEFAULT CURRENT_DATE NOT NULL,
    requests_used integer DEFAULT 0 NOT NULL,
    requests_limit integer DEFAULT 7500 NOT NULL,
    requests_remaining integer GENERATED ALWAYS AS ((requests_limit - requests_used)) STORED,
    last_updated timestamp with time zone DEFAULT now() NOT NULL
);

To je přesně to, co potřebuješ.

Každý sport má:

den

kolik requestů už bylo použito

kolik zbývá

🔟 Jak by měl scheduler rozhodovat

Například takto:

for sport in sports_import_plan order by priority:

    budget = remaining_requests(sport)

    if budget <= 0:
        continue

    jobs = ingest_planner
           where sport = sport
           and status = 'pending'
           order by priority

    for job in jobs:

        if estimated_cost(job) <= budget:
            push scheduler_queue
            budget -= estimated_cost
